# 股票技术指标交互工具 Spec

## 1. 目标

新增一个独立网页工具，用于对任意股票日线数据生成类似 `task2/question3` 的技术指标图表和基础解析。

工具支持两种数据来源：

1. 本地导入：用户上传本地 CSV 文件。
2. 动态获取：用户输入股票代码、市场类型和 MCP server 等参数，由本地后端通过 MCP 或数据接口获取近三年日线数据。

本文件最初为执行前 spec；当前已按确认点实现静态 HTML、本地后端、配置示例和 README。真实 MCP 联调通过后端配置文件 `stock_indicator_tool_config.json` 接入。

## 2. 输出目录

所有工具相关产物统一放在：

- `task2/tool/`

建议后续生成：

- `stock_indicator_tool_spec.md`：本 spec。

## 3. 总体架构

建议采用“前端网页 + 可选本地后端”的结构。

### 3.1 前端职责

前端负责：

- 本地 CSV 文件选择和读取。
- 左侧操作栏展示。
- 指标参数设置。
- 数据校验结果展示。
- 图表动态渲染。
- 指标基础解析展示。
- 错误提示和停止执行状态展示。

### 3.2 后端职责

动态下载股票数据不能只靠静态 HTML 完成，原因：

- 浏览器页面不能直接安全地调用本地 MCP server。
- MCP server 参数、token、命令路径等不应暴露在纯前端。
- 动态下载需要处理超时、接口异常、股票代码错误、市场识别错误和返回字段校验。

因此，如需支持“通过股票代码动态下载近三年数据”，建议提供本地后端：

- Python FastAPI/Flask 或标准库 HTTP server。
- 接收前端请求。
- 根据市场类型和代码调用 MCP server 或指定数据源。
- 返回统一格式 JSON。
- 对错误情况返回结构化错误信息。

如果用户只使用本地 CSV 导入，静态 HTML 即可运行；如果使用动态下载，则必须启动后端。

网页采用左右两栏。

左侧较窄，建议宽度约 300-360px，固定在页面左侧。

- 动态获取区域：

  - 股票代码输入。
  - 中文名称补全显示。
  - MCP server URL 或名称。
  - MCP tool 名称或数据源标识。
  - 超时时间设置，默认 30 秒。
  - 获取按钮。
- 指标参数设置：
  - RSI 周期，默认 6、14、24。
  - MACD 快线、慢线、信号线，默认 12、26、9。
  - Bollinger Bands 周期和标准差倍数，默认 20、2。
  - ATR 周期，默认 14。
  - ADX 周期，默认 14。
  - BIAS 周期，默认 6、12、24。
  - Z-score 周期，默认 20。

## 5. 数据输入格式

### 5.1 标准字段

工具内部统一使用以下字段：

- `ts_code`：股票代码。
- `trade_date`：交易日期。
- `open`：开盘价。
- `high`：最高价。
- `low`：最低价。
- `close`：收盘价。
- `pre_close`：前收盘价。
- `change`：涨跌额。
- `pct_chg`：涨跌幅。
- `vol`：成交量。
- `amount`：成交额。

### 5.2 本地 CSV 兼容

本地 CSV 导入时，优先识别上述字段。

如果用户文件字段名不同，建议支持有限的字段映射，例如：

- `date` / `datetime` / `tradeDate` -> `trade_date`
- `volume` -> `vol`
- `turnover` / `value` -> `amount`

如果缺少核心字段，则停止执行并报错。

核心字段：

- `trade_date`
- `open`
- `high`
- `low`
- `close`
- `vol`

`pre_close`、`change`、`pct_chg` 可以在缺失时通过 `close.shift(1)` 和 `close` 计算补齐，但必须在报告中说明。

## 6. 动态下载设计

### 6.1 市场区分

用户必须选择市场：

- A股。
- 港股。

市场选择会影响：

- 股票代码格式校验。
- 数据源接口调用参数。
- 股票中文名补全方式。
- 交易日历差异说明。
- 成交量和成交额单位说明。

### 6.2 代码格式建议

A股：

- 支持 `688981.SH`、`600000.SH`、`000001.SZ` 等格式。
- 若用户只输入 6 位数字，可根据规则辅助提示，但不应盲目推断。

港股：

- 支持 `00981.HK` 等格式。
- 若用户输入 `981`，可提示补全为 `00981.HK`，但需用户确认或明确展示补全结果。

### 6.3 中文名补全

动态下载时应补全股票中文名：

- 如果 MCP 或数据接口返回股票名称，则直接使用。
- 如果只返回代码，则后端应尝试调用证券基础信息接口补全。
- 如果无法补全，则显示“未知名称”，但不阻断指标计算。

本地 CSV 导入时：

- 若数据中存在 `name`、`stock_name`、`security_name` 字段，则读取。
- 若不存在，则根据 `ts_code` 尝试从内置映射或后端接口补全。
- 若仍无法补全，则显示文件名或“未知名称”。

### 6.4 MCP 参数

前端设置项建议包括：

- MCP server 名称或 URL。
- MCP tool 名称。
- API token 或认证方式，尽量只保存在本地后端配置中，不直接暴露在前端。
- 超时时间。
- 重试次数，默认 0 或 1。

后端返回统一 JSON：

```json
{
  "ok": true,
  "market": "A股",
  "ts_code": "688981.SH",
  "name": "中芯国际",
  "source": "mcp",
  "rows": [],
  "warnings": []
}
```

错误返回：

```json
{
  "ok": false,
  "error_code": "TIMEOUT",
  "message": "动态获取数据超时，请检查 MCP server 或网络连接。",
  "details": {}
}
```

## 7. 数据校验和停止规则

无论本地导入还是动态获取，都必须先校验数据。

### 7.1 完整性校验

检查：

- 是否为空文件或空返回。
- 是否存在核心字段。
- 日期是否可解析。
- 数值字段是否可转换为数字。
- 核心字段是否有缺失值。
- 是否至少有足够样本计算最长窗口指标。

建议最少样本数：

- 至少 60 个交易日。
- 如果不足 60 个交易日，停止执行并提示样本不足。

### 7.2 合理性校验

检查：

- `high >= low`。
- `open/high/low/close > 0`。
- `vol >= 0`。
- `amount >= 0`。
- 日期不能重复。
- 日期应按交易日升序排列。
- `close` 不能全部相同。
- 不应存在无限值。
- `change` 如存在，应接近 `close - pre_close`。
- `pct_chg` 如存在，应接近 `change / pre_close * 100`。

### 7.3 错误处理规则

如果发现严重错误：

- 停止执行。
- 不生成图表。
- 右侧显示错误卡片。
- 左侧状态区显示失败原因。

严重错误包括：

- 无法读取文件。
- MCP 请求超时。
- 股票代码错误。
- 市场类型与代码不匹配。
- 数据为空。
- 核心字段缺失。
- 核心字段缺失值无法处理。
- 价格字段存在非正数。
- 日期重复且无法自动去重。
- 样本数量不足。

如果发现轻微问题：

- 可以继续执行，但必须展示 warning。

轻微问题包括：

- `pre_close` 缺失但可用前一日收盘价补齐。
- `change` 缺失但可计算。
- `pct_chg` 缺失但可计算。
- 股票中文名无法补全。

## 8. 指标计算

指标应与 `task2/question3` 保持一致。

默认指标：

- RSI：6、14、24。
- MACD：12、26、9。
- Bollinger Bands：20 日，2 倍标准差。
- ATR：14。
- OBV：含 20 日 OBV 均线。
- ADX：14。
- BIAS：6、12、24。
- Z-score：20。

左侧参数调整后，右侧图表应重新计算。

## 9. 关键事件高亮

各图关键点应一目了然。

建议高亮事件：

- Bollinger Bands：
  - 收盘价突破上轨。
  - 收盘价跌破下轨。
- RSI：
  - RSI14 > 70。
  - RSI14 < 30。
- MACD：
  - DIF 上穿 DEA：金叉。
  - DIF 下穿 DEA：死叉。
- ATR：
  - ATR/收盘价处于样本前 10% 高波动区间。
- OBV：
  - 疑似顶背离。
  - 疑似底背离。
- ADX：
  - ADX 上穿 25。
  - +DI / -DI 方向切换。
- BIAS：
  - BIAS12 处于样本前 10%。
  - BIAS12 处于样本后 10%。
- Z-score：
  - Z-score > 2。
  - Z-score < -2。

高亮点应支持 hover，显示：

- 日期。
- 指标值。
- 事件名称。
- 简短解释。

## 10. 图表交互

图表要求与 question3 一致：

- legend 可点击隐藏/显示对应曲线。
- hover 显示日期和数值。
- 图表应避免横坐标标签溢出。
- 图表容器自适应右侧宽度。
- 页面不出现整体横向滚动条。
- 宽表或参数表如溢出，应在内部滚动。

## 11. 基础解析

右侧每张图下方生成一段基础解析。

解析内容包括：

- 当前最新交易日指标状态。
- 样本中关键事件次数。
- 当前指标是否偏趋势、震荡、超买、超卖、高波动。
- 对策略构造的启示。

示例：

- MACD：当前 DIF 是否高于 DEA，金叉/死叉次数，是否需要用 ADX 过滤震荡期假信号。
- RSI：当前 RSI14 是否接近超买/超卖，是否适合均值回复观察。
- ATR：当前波动水平是否高，是否建议降低仓位或放宽止损。
- OBV：价格上涨是否有成交量支持，是否出现背离线索。

## 12. 技术实现建议

### 12.1 前端

建议使用纯 HTML/CSS/JavaScript 实现，便于本地打开。

图表可以延续 question3 中的原生 SVG 实现：

- 不依赖外部 CDN。
- 不依赖联网。
- legend 交互可控。
- 适合离线作业提交。

### 12.2 后端

动态下载建议使用 Python 后端：

- 标准库 `http.server` 或 Flask/FastAPI。
- 路由：
  - `GET /health`
  - `POST /fetch_daily`
  - `POST /lookup_name`
- 超时控制：
  - 默认 30 秒。
  - 前端可设置。
- 错误类型：
  - `TIMEOUT`
  - `INVALID_CODE`
  - `INVALID_MARKET`
  - `MCP_UNAVAILABLE`
  - `EMPTY_DATA`
  - `SCHEMA_ERROR`
  - `DATA_VALIDATION_ERROR`

### 12.3 MCP 接入边界

MCP server 的具体调用方式应独立封装在后端函数中。

建议接口：

```python
def fetch_daily_from_mcp(market: str, ts_code: str, start_date: str, end_date: str, config: dict) -> dict:
    ...
```

这样未来无论接入 Tushare、AkShare、自定义 MCP，前端逻辑都不用改。

## 13. 安全和可用性

注意：

- 不要在前端 HTML 中硬编码 token。
- 不要把 MCP 凭据写入报告。
- 后端只监听本地地址 `127.0.0.1`。
- 超时和错误要清晰显示。
- 如果动态获取不可用，本地 CSV 导入仍应可用。
